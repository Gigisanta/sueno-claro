# Financial model

## Model stance

This is an asset-style product, not a guaranteed venture-scale bet. The right early metric is low-cost validation: organic traffic, repeat usage, and willingness to pay for convenience.

## Revenue streams

1. Web contextual ads after result.
2. Affiliate/sponsor blocks on educational pages, carefully separated from the tool.
3. iOS premium via RevenueCat.
4. Later: web paid links via RevenueCat Web Purchase Links if cross-platform premium matters.

## Conservative scenarios

Assumptions:

- Free web sessions are the main traffic source.
- Core calculator remains free.
- Premium annual net modeled at 70% platform net on $9.99/year.
- Ads modeled separately with low/medium RPM.

## Interpretation

The web product needs real SEO volume to matter. Premium alone will be modest unless widgets/reminders create repeat value. Keep costs near zero until traffic proves distribution.

## Unit economics to monitor

- Hosting cost per 100k sessions.
- Ad RPM after result.
- Premium conversion from app users.
- Refund rate.
- Support burden per paying user.
- Content production cost per indexed page.

## Kill/pivot criteria

- No organic impressions after 60-90 days despite technical SEO pass.
- Calculator completion < 40% after UX fixes.
- Ad revenue harms engagement beyond acceptable threshold.
- Premium conversion < 0.2% after app convenience features are live.
